# Project Plan – Media Buying Dashboard

---

## 1. Executive Summary
The Media Buying Dashboard provides a unified view of advertising performance by aggregating market-intelligence data from **10 diverse external sources** (pytrends, eBay, Reddit, X API, Meta Ads Library, Yelp Fusion, Foursquare Places, Microsoft Bing Webmaster, Skyscanner, Indeed/Adzuna) and processing it through a **4-stage ETL pipeline** (Source Ingestion → Sector Grouping → Company/Platform Grouping → KPI Building) to surface the highest-performing media-buying opportunities across commerce sectors (Technology, Finance, Manufacturing, Retail, Health & Wellness, Travel, Job Market, …). It surfaces a **Top Opportunity** card based on a composite score of six key performance indicators (ROAS, CAC, CLTV, Conversion Rate, Scalability, Attribution Accuracy) and a hierarchical drill‑down UI.

---

## 2. Project Objectives
| Objective | Success Metric |
|----------|----------------|
| Deliver a production‑ready dashboard | MVP released to internal users within 12 weeks |
| Enable data‑driven media‑buy decisions | Top Opportunity card updates within 2 seconds; drill‑down ≤ 1 second |
| Ensure data freshness | KPI data refreshed ≤ 15 minutes, staleness indicator shown when older |
| Provide extensible architecture for future platforms | Modular monolith with clear extension points (documented in HLD/LLD) |
| Achieve high usability | SUS ≥ 80 in usability testing |

---

## 3. Scope
- Platforms: 10 diverse market-data sources (pytrends, eBay, Reddit, X API, Meta Ads Library, Yelp Fusion, Foursquare Places, Bing Webmaster, Skyscanner, Indeed/Adzuna). The deprecated 5 ad-platform APIs (Google Ads, Meta Ads, TikTok, LinkedIn, iHeart Radio) have been replaced and marked `@Deprecated` per ADR-0007 (Spring Events over Kafka).
- Commerce sectors: Technology, Finance, Manufacturing, Retail, Health & Wellness, etc.
- Core features: Top Opportunity card, hierarchical platform‑sector view, KPI table with 13 columns, global sector filter, CSV export, ROI Calculator, role‑based KPI masking, automatic data‑refresh and staleness icons.
- Non‑functional: performance, scalability, security, observability, compliance (GDPR/SOC‑2).

**Key Infrastructure Architecture Decisions** (see HLD §14 and LLD §18–21):
1. **EKS** over ECS — supports Helm, HPA, IRSA, and future microservices migration.
2. **Tiered VPC subnets** (public / private-app / private-data) — defense-in-depth for RDS/Kafka/Redis.
3. **Kafka topic redesign** — 3 topics (`kpi.raw`, `kpi.refresh`, `data-refresh.internal`) instead of 2 for cleaner producer/consumer decoupling.
4. **External Secrets Operator** — syncs AWS Secrets Manager → K8s Secrets without baking credentials into Helm values.
5. **Canary deployment strategy** — 10% traffic → observe → full rollout, with `helm rollback` on failure.

**Architecture Decisions for Integration Wiring** (see also HLD §9 and LLD §7):

| Decision | Rationale | Trade-Off |
|---|---|---|
| **Single `AdPlatformIngestionScheduler`** over 5 separate `@Scheduled` methods | One place to manage scheduling, timeout, error aggregation, and monitoring. Adding a new platform requires only adding a new `fetchAndPublish()` call — no new `@Scheduled` method. | Single class grows with platform count. Mitigated by extracting per-platform logic into the wrappers themselves. |
| **`@Async` + `CompletableFuture.allOf()`** over sequential execution | Parallelism is essential: 5 platforms × 30s worst-case API call = 150s sequential vs. 30s parallel (with timeout). One platform's slowness or failure does not block others. | Requires `@EnableAsync` and thread pool tuning (`spring.task.execution.pool.core-size=10`). `CompletableFuture.get()` with timeout prevents indefinite blocking. |
| **`KpiNormalizer` as separate component** over putting mapping inside `AdPlatformIngestionScheduler` | Separation of concerns: the scheduler orchestrates; the normalizer maps. Unit‑testable in isolation. Easy to extend if new fields are added to `RawKPIEvent`. | One more small class. Negligible overhead. |
| **Rate limiter in `BaseApiWrapper`** over in `AdPlatformIngestionScheduler` | The wrapper is the correct place because it knows the platform's rate limit. The scheduler just fires — the wrapper self‑throttles. If we ever add direct wrapper calls outside the scheduler (e.g., admin debug endpoint), rate limiting is still enforced. | Wrapper must hold a reference to `AdPlatformRateLimiter`. |
| **`ConcurrentHashMap` + synchronized refresh** for token cache (not Redis) | Token cache is small (5 entries), accessed frequently, and must survive only the JVM lifetime. A JVM‑local cache avoids the Redis network hop on every API call. Synchronized per‑platform (not global) prevents concurrent token refresh storms while allowing other platforms to proceed. | Tokens are lost on pod restart. Acceptable — the next API call will transparently re‑authenticate. |
| **Mock response for iHeart Radio** over blocking on missing public API | iHeart Media does not publish a full advertising metrics API. Returning realistic mock data with `dataSource="iHeart Radio (estimated)"` allows the pipeline to be complete for MVP while being honest about data provenance. Can be replaced with a real integration when available. | Mock data may diverge from real performance. Mitigated by the `dataSource` label and future integration path. |
| **`PlatformApiResponse` (BigDecimal) → `RawKPIEvent` (Double)** intermediary mapping | Wrapper output uses `BigDecimal` for precision (financial data). Kafka messages use `Double` for JSON serialization simplicity. The normalizer handles the conversion with null‑safety. | Potential precision loss in the Double conversion (negligible for dashboard display; the original `BigDecimal` values in PostgreSQL retain full precision). |

### 5.6 Data Source Architecture Migration (10-Source Pipeline) — PLANNED (Revised: Kafka Removed)

> **Goal**: Replace the 5 direct ad-platform API wrappers (Google Ads, Meta Ads, TikTok, LinkedIn, iHeart Radio) with a **4-stage ETL pipeline** ingesting from **10 diverse market-data sources**: pytrends, eBay, Reddit, X API (Twitter), Meta Ads Library, Yelp Fusion, Foursquare Places, Microsoft Bing Webmaster, Skyscanner, and Indeed/Adzuna. ## 7. Risk Register & Mitigation
| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| External ad‑platform API changes | High – data pipeline breakage | Medium | Wrapper abstraction with circuit‑breaker; versioned API clients; early integration tests |
| Pipeline accumulator race condition (Spring Events) | Medium – duplicate/corrupted KPI values | Low | `ConcurrentHashMap` with `computeIfAbsent` + synchronized list append; atomic batch drain (iterator removal); idempotent PostgreSQL upserts mitigate duplicates |
| Composite scoring weights mis‑configured | Medium – wrong Top Opportunity shown | Low | Admin UI with validation; unit tests for scoring algorithm; audit logs of weight changes |
| Performance under 10 k concurrent users | High – UI becomes sluggish | Medium | Horizontal pod autoscaling; Redis cache for scoring and KPI queries; load‑testing before go‑live |
| Security breach exposing PII | Critical | Low | TLS everywhere, RBAC, audit logging, regular pen‑test, secret rotation |
